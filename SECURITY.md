# 🛡️ Security Policy

We take the overall security, accuracy, and operational efficiency of our community challenges seriously. If you encounter any structural vulnerabilities, logical layout flaws, or calculation bugs, please review the guidelines below to report them securely.

---

## 📦 Supported Project Versions

We actively monitor, patch, and maintain only the latest modules deployed on the `main` branch. Older commits or independent branches are not systematically tracked.

| Version | Deployed Status | Maintenance Support |
| :---: | :---: | :--- |
| **1.0.x** | 🚀 Production | 🟢 Fully Supported & Actively Monitored |
| **< 1.0** | 📉 Legacy / Archive | ❌ No Longer Maintained |

---

## 🚨 Reporting a Vulnerability

> ### ⚠️ Crucial Step: Do Not Open a Public Issue
> If you detect a fatal loop flaw, broken challenge parameter, or a syntax vulnerability, **please do not open a public GitHub issue or pull request.** Public exposure allows errors to be copied across the community workspace before a patch can be integrated.

Please report your discovery privately through our direct channels:

* 📧 **Secure Email Portal:** [contact@codexjava.com](mailto:contact@codexjava.com)
* 📸 **Direct Instagram Communication:** [@codexjava_](https://instagram.com/codexjava_)

### 📝 What to Include in Your Report
To help us triage and deploy a quick structural patch, please provide:
1. A clear description of the logical flaw or validation bug.
2. The exact module folder location (e.g., `Day-01_Introduction/`).
3. A short code snippet or step-by-step logic demonstrating the issue.
